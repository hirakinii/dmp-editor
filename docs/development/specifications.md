## 1. システム構成・アーキテクチャ

### 1-1. 概要

DMP Editor は、公的研究資金の提供機関が要求するデータマネジメントプラン（DMP）を作成・編集・提出形式に変換するための **Web アプリケーション**である。バックエンドサーバーを持たない**フロントエンド専用の SPA（Single Page Application）**として設計されており、DMP データの永続化は外部サービス（GakuNin RDM）に委譲している。

### 1-2. デプロイ構成

| 環境 | 方式 | URL |
|------|------|-----|
| 本番環境 | GitHub Pages | `https://nii-dg.github.io/dmp-editor/` |
| セルフホスト | Docker（Nginx）| `http://localhost:3000` |
| 開発環境 | Docker + Vite Dev Server | `http://localhost:5173` |

### 1-3. 技術スタック

| カテゴリ | 採用技術 |
|----------|----------|
| UI フレームワーク | React 18 + TypeScript |
| UI コンポーネント | MUI（Material UI v7）+ MUI X Tree View v8 |
| ルーティング | React Router v7 |
| フォーム管理 | react-hook-form + Zod |
| 非同期データフェッチ | TanStack Query v5 |
| グローバル状態管理 | Recoil（認証トークン） |
| 多言語対応 | i18next + react-i18next |
| ファイル操作 | fflate（JSPS テンプレートの ZIP/XML 直接編集） |
| ビルドツール | Vite |
| テスト | Vitest + Testing Library |

### 1-4. 外部システム連携

```
┌─────────────────────────────────────────────────────┐
│                   DMP Editor (SPA)                   │
│                  ブラウザ上で動作                      │
│                                                     │
│  ┌──────────────┐   ┌─────────────┐   ┌──────────┐ │
│  │  GRDM Client  │   │ KAKEN Client │   │ ROR Client│ │
│  └──────┬───────┘   └──────┬──────┘   └────┬─────┘ │
└─────────┼─────────────────┼───────────────┼────────┘
          │                 │               │
          ▼                 ▼               ▼
  ┌──────────────┐  ┌─────────────┐  ┌──────────────┐
  │ GakuNin RDM  │  │  KAKEN API  │  │   ROR API    │
  │  API v2      │  │ (NII CiNii) │  │ (ror.org)    │
  │(認証・DMP保存)│  │(科研費情報) │  │(機関検索)    │
  └──────────────┘  └─────────────┘  └──────────────┘
```

| 外部システム | 目的 | 認証方式 |
|------------|------|---------|
| GakuNin RDM (GRDM) API v2 | DMP ファイルの読み書き・プロジェクト管理 | Personal Access Token（ブラウザの LocalStorage に保存） |
| KAKEN API（NII CiNii） | 科研費課題番号による DMP プロジェクト情報の自動補完 | Application ID（任意・環境変数 `VITE_KAKEN_APP_ID`） |
| ROR（Research Organization Registry）API | データ管理機関のコード（ROR ID）検索 | 不要（公開 API） |

### 1-5. データ永続化

DMP データは、GRDM のプロジェクト上のファイルストレージ（osfstorage）に **`dmp-project.json`** として保存される。アプリ本体はデータベースを持たない。

```
GRDM プロジェクト（DMP-{プロジェクト名}）
  └── osfstorage/
        └── dmp-project.json   ← DMP データ本体（JSON 形式）
```

### 1-6. 画面遷移

```
[起動]
  └─→ 認証画面（GRDM Token 入力）
        └─→ プロジェクト一覧画面
              ├─→ DMP 編集画面（新規作成）  ← 5ステップ Stepper フォーム
              │     └─→ プロジェクト一覧画面（保存 / キャンセル / エクスポート）
              ├─→ DMP 編集画面（既存編集）  ← 5ステップ Stepper フォーム
              │     └─→ プロジェクト一覧画面（保存 / キャンセル / エクスポート）
              ├─→ DMP 詳細表示画面（読み取り専用）
              └─→ エラー画面（404 / 予期しないエラー）
```

編集画面の 5 ステップ：

| ステップ | 内容 |
|---------|------|
| 1 | DMP メタデータ（種別・提出日・研究フェーズ） |
| 2 | プロジェクト情報（資金配分機関・課題番号・KAKEN 自動補完） |
| 3 | 担当者情報（複数登録・並べ替え・GRDM ユーザー検索） |
| 4 | 研究データ情報（データセット複数登録・GRDM ファイルリンク・ROR 検索） |
| 5 | GRDM 連携（プロジェクト関連付け・ファイルツリーマッピング） |

## 2. 提供機能

### 2-1. 認証・セッション管理

| 機能 | 説明 |
|------|------|
| **GRDM トークン認証** | ユーザーが GRDM のパーソナルアクセストークンを入力し、GRDM API との疎通を確立する |
| **トークンのローカル保存** | 入力されたトークンはブラウザの LocalStorage のみに保存し、サーバーには送信しない |
| **ログインユーザー情報の表示** | 認証後、GRDM からユーザー情報（氏名・所属・ORCID・e-Rad 番号）を取得しヘッダーに表示する |

### 2-2. DMP プロジェクト管理

| 機能 | 説明 |
|------|------|
| **プロジェクト一覧表示** | GRDM 上の `DMP-` プレフィックス付きプロジェクトを一覧表示する（プロジェクト名・作成日・最終更新日） |
| **新規 DMP プロジェクト作成** | GRDM 上に `DMP-{プロジェクト名}` の形式で新規プロジェクトを作成し、空の `dmp-project.json` を初期化する |
| **既存 DMP の編集** | 既存プロジェクトの `dmp-project.json` を読み込み、編集フォームに展開する |
| **DMP 詳細表示（読み取り専用）** | 編集モードを経ずに DMP の内容を閲覧するための読み取り専用ビューを提供する |
| **GRDM プロジェクトページへのリンク** | プロジェクト一覧から GRDM 本体の該当プロジェクトページへ直接遷移できる |

### 2-3. DMP 内容の編集

DMP の構造は以下の4つのセクションからなる。

#### (a) DMP メタデータ

| フィールド | 説明 |
|-----------|------|
| 種別 | 新規 / 修正 / 更新 |
| 提出日 | DMP を資金配分機関へ提出する日付 |
| DMP 作成年月日 / 最終更新年月日 | 自動管理 |
| 研究フェーズ | 計画時 / 研究中 / 報告時 |

#### (b) プロジェクト情報

| フィールド | 説明 |
|-----------|------|
| 資金配分機関情報 | 資金配分機関名・プログラム名・プログラムコード（NISTEP 参照） |
| 体系的番号（NISTEP） | 15桁の体系的番号 |
| プロジェクト名 | 研究課題名 |
| 採択年度・開始年度・終了年度 | 研究期間 |
| **KAKEN 番号による自動補完** | 科研費課題番号を入力すると KAKEN API から資金配分機関・プログラム名・プログラムコード・採択/開始/終了年度を自動取得し、研究者情報（担当者セクション）も同時に取り込む。取得内容は **KakenConfirmDialog**（プログラム名・プロジェクト名・採択年度・研究者リストを表示）で確認後に適用する。同姓同名・同所属の担当者が既存登録と重複する場合は **DuplicatePersonDialog** で個別にスキップ可能 |

#### (c) 担当者情報（複数登録・並べ替え対応）

| フィールド | 説明 |
|-----------|------|
| 役割 | 研究代表者 / 研究分担者 / 管理対象データの作成者 / 管理対象データの管理責任者（複数選択可） |
| 氏名・所属機関 | 担当者の基本情報（必須） |
| e-Rad 研究者番号 | 府省共通研究開発管理システムの研究者 ID |
| ORCID | 国際的な研究者識別子 |
| 連絡先（メールアドレス） | 任意 |
| GRDM ユーザー ID | GRDM アカウントとの紐付け用（任意） |

**追加機能:**

| 機能 | 説明 |
|------|------|
| **GRDM ユーザー検索パネル** | 姓で GRDM 登録ユーザーを検索し、氏名・所属・ORCID・e-Rad 番号・GRDM ユーザー ID を一括取り込む |
| **値の出典追跡** | 各フィールドの取得元（KAKEN / GRDM / ユーザー手入力）をフィールドごとにチップで表示する |
| **制約チェック** | 「研究代表者」および「管理対象データの管理責任者」は 1 人のみ登録可能。同姓同名・同所属の重複登録を防止する |

#### (d) 研究データ情報（複数登録対応）

**基本情報**

| フィールド | 説明 |
|-----------|------|
| 管理対象データの名称 | データセットの識別名（必須） |
| 掲載日・掲載更新日 | データセットの掲載日または更新日（研究中・報告時フェーズでは必須） |
| データの説明 | 内容・概要（必須） |
| データの分野 | 11 分野から選択（ライフサイエンス・情報通信 等）（必須） |
| データ種別 | 14 種別から選択（実験データ・観測データ 等）（必須） |
| 概略データ量 | 任意入力 |
| 取得・収集方法 | 任意入力 |
| 再利用を可能にするための情報 | 任意入力 |

> **動的バリデーション**: 「掲載日・掲載更新日」は DMP メタデータの研究フェーズが **「研究中」または「報告時」** の場合に必須フィールドとなる。「計画時」では任意入力。

**セキュリティ・倫理ポリシー**

| フィールド | 説明 |
|-----------|------|
| 機微情報の有無 | 有 / 無 |
| 機微情報がある場合の取扱い方針 | 個人情報・機密情報の管理方針を記述 |
| 管理対象データの利活用・提供方針（研究活動時） | 研究期間中のアクセス・利用ポリシー |

**保存・公開計画**

| フィールド | 説明 |
|-----------|------|
| リポジトリ情報（研究活動時） | 研究期間中の保存先 |
| バックアップ場所 | バックアップ先 |
| アクセス権 | 公開 / 共有 / 非共有・非公開 / 公開期間猶予 |
| 公開予定日 | データ公開の予定日 |
| 公開・提供方針詳細 | 公開の詳細条件 |
| リポジトリ情報（研究活動後）| 研究終了後の保存先（リポジトリ URL・DOI） |
| 研究データの保存場所・保存期間（研究事業終了後） | 長期保存の計画 |

**管理体制**

| フィールド | 説明 |
|-----------|------|
| データ管理機関 | 機関名（**ROR 検索**による機関コード取得をサポート）（必須） |
| データ管理機関コード（ROR ID） | 国際的な機関識別子 |
| データ管理者 | 担当部署（必須） |
| データ管理者の連絡先 | 担当部署の連絡先（必須） |
| 管理対象データの作成者 | 担当者情報との紐付け（担当者セクションの登録者をドロップダウンで選択） |
| 研究データの保存場所（研究事業終了後） | 長期保存の計画 |
| 研究データの保存期間（研究事業終了後） | 長期保存の計画 |

**GRDM ファイルリンク**

| 機能 | 説明 |
|------|------|
| **GRDM ファイルの選択** | **GrdmSelectModal** でデータセットに紐付ける GRDM ファイルをツリー形式で選択する（複数選択可） |
| **GRDM メタデータの比較・採用** | **GrdmCompareModal** で、リンク済みファイルの GRDM 上のメタデータ（名称・分野・種別・アクセス権等 13 フィールド）と現在の入力値を並べて比較し、GRDM 側の値を選択的に取り込める |
| **値の出典追跡** | GRDM から取り込んだフィールドは出典チップ（GRDM）で識別できる |

### 2-4. GRDM 連携機能

| 機能 | 説明 |
|------|------|
| **GRDM プロジェクトの関連付け** | DMP プロジェクトに対し、実データが格納されている GRDM プロジェクトをステップ 5 のテーブルで紐付ける |
| **ファイルツリー参照** | ステップ 5 で、関連付けられた GRDM プロジェクトのファイル・フォルダ構造をツリー形式で表示する。各データセット定義に対して Link / Unlink ボタンでファイルを紐付けできる |
| **研究データ情報画面からのファイル選択** | ステップ 4（研究データ情報）の各データセット編集画面から **GrdmSelectModal** を開き、ファイルツリーで直接リンクするファイルを選択できる |
| **DMP データ項目とファイルのマッピング** | 各データセット定義に対して、実際の GRDM ファイルを紐付ける（計画と実態のトレーサビリティ確保）。フォルダを選択した場合は全子ファイルを自動リンクする |
| **ファイルメタデータの記録** | 紐付けたファイルのパス・サイズ・MD5/SHA256 ハッシュ値・作成日・更新日・GRDM リンク URL を DMP JSON 内に保存する |

### 2-5. DMP エクスポート

| エクスポート形式 | 説明 |
|---------------|------|
| **汎用 Excel 形式（.xlsx）** | DMP の全フィールドを 4 シート（DMP情報・プロジェクト情報・担当者情報・研究データ情報）に展開してエクスポートする |
| **JSPS 様式 Excel 形式（.xlsx）** | 日本学術振興会（JSPS）の DMP 提出様式テンプレート（`jsps_template.xlsx`）を ZIP/XML レベルで直接編集して DMP データを埋め込む。担当者・データ項目数が様式のデフォルト行数（担当者 6 行・データ項目 5 行）を超える場合は自動的に行を挿入・スタイルコピーして拡張する |

### 2-6. UI 補助機能

| 機能 | 説明 |
|------|------|
| **5ステップ Stepper フォーム** | 編集画面を 5 つのステップに分割し、ステップ移動前に必須項目バリデーションを実施する |
| **入力例（プレースホルダー）の表示** | 各フィールドに具体的な入力例を表示し、記述内容のガイドとする |
| **ヘルプアイコン（ツールチップ）** | 専門用語・各項目の定義をツールチップで説明する |
| **値の出典追跡チップ** | 担当者・研究データ情報の各フィールドに、値の取得元（KAKEN / GRDM / ユーザー入力）をフィールド単位でチップ表示する |
| **動的必須フィールド** | 研究フェーズ（計画時 / 研究中 / 報告時）に応じてフィールドの必須/任意が切り替わる |
| **未保存変更の警告** | フォームに未保存の変更がある状態でページを離脱しようとすると警告ダイアログを表示する |
| **トースト通知** | 保存成功・エラー等の操作結果をスナックバー（トースト）で通知する |
| **多言語対応** | i18next による日本語・英語の UI 切り替えをサポートする |
| **バージョン情報の表示** | フッターにアプリバージョンを表示し、不具合報告時の特定を容易にする |
| **エラーバウンダリ** | 予期しないエラーを捕捉し、ユーザーにわかりやすいエラー画面を表示する |

## 3. 各機能が解決しようとしている研究データ管理上の課題

### 課題 1: DMP 作成に要する労力・専門知識の壁

**課題**: DMP は資金配分機関ごとに異なる様式・項目があり、不慣れな研究者にとって何をどう書けばよいか不明瞭である。
**対応機能**:
- 入力例（プレースホルダー）によるガイダンス
- 各項目のヘルプアイコン（ツールチップ）
- 必須項目のアスタリスク表示
- JSPS 様式への直接エクスポート（提出フォーマットとの乖離をゼロにする）

---

### 課題 2: プロジェクト情報の手動入力の手間・転記ミス

**課題**: 科研費課題番号などのプロジェクト情報は既存データベースに存在するが、DMP に手動転記する必要があり、ミスや工数が発生する。
**対応機能**:
- **KAKEN 番号による自動補完**（課題番号入力 → 資金配分機関・プログラム・年度を自動取得）
- ログインユーザーの GRDM プロファイル情報（氏名・所属・ORCID・e-Rad番号）を研究代表者情報として自動設定

---

### 課題 3: DMP の計画（Plan）と実際のデータ管理（Do）の乖離

**課題**: DMP は計画書として作成されるが、その後の実際のデータ管理（どのシステムにどのファイルが存在するか）との対応が管理されず、DMP が形骸化する。
**対応機能**:
- **GRDM プロジェクトとの関連付け**（DMP と実データストレージの紐付け）
- **ファイルツリーの参照と DMP データ項目へのマッピング**（計画データ定義と実ファイルのトレーサビリティ確保）

---

### 課題 4: 研究データの機微性・倫理的配慮の記録漏れ

**課題**: 個人情報・機密情報を含むデータの取扱い方針が明文化されず、法令・倫理指針への対応が不透明になる。
**対応機能**:
- データセットごとの**機微情報の有無と取扱い方針**の記録フィールド
- アクセス権・公開予定日の設定による公開計画の明確化

---

### 課題 5: 研究データのオープンサイエンス方針への非対応

**課題**: 資金配分機関のオープンサイエンス方針に従い、データの公開計画（いつ・どこで・どのような条件で）を具体的に示す必要があるが、管理が煩雑である。
**対応機能**:
- アクセス権（公開 / 共有 / 非共有・非公開 / 公開期間猶予）の選択
- 公開予定日の設定
- 研究活動後のリポジトリ情報・DOI リンクの記録
- 研究データの保存期間の明示

---

### 課題 6: 研究者・機関の識別子管理の不統一

**課題**: 同姓同名の研究者の識別、国際連携における機関情報の統一表記が困難である。
**対応機能**:
- 担当者情報への **e-Rad 研究者番号・ORCID** の登録
- **ROR ID 検索**によるデータ管理機関の標準的識別子（ROR ID）の取得・記録

---

### 課題 7: DMP の提出形式への変換コスト

**課題**: 編集中の DMP データを資金配分機関の定める提出様式（Excel等）に変換する作業が手動で煩雑である。
**対応機能**:
- **JSPS 様式 Excel エクスポート**（テンプレートへのデータ自動埋め込み・行数の自動拡張）
- **汎用 Excel エクスポート**（機関横断的な記録・共有・バックアップ用）

---

### 課題 8: DMP データのセキュリティとポータビリティ

**課題**: 認証情報がサーバーに保存されると漏洩リスクが生じる。また、DMP データが特定のクローズドシステムに閉じ込められると可搬性が失われる。
**対応機能**:
- GRDM トークンの**ブラウザ LocalStorage のみへの保存**（サーバー非送信）
- DMP データの **JSON ファイルとして GRDM に保存**（可搬性・バックエンドサーバー不要のアーキテクチャ）

---

### 課題 9: 研究チーム内での DMP の共同管理

**課題**: 複数の担当者（研究代表者・分担者・データ管理責任者）が存在する研究プロジェクトにおいて、役割に応じた記録・更新が必要である。
**対応機能**:
- 担当者情報の**複数登録・役割付与・並べ替え**機能
- DMP の保存先を GRDM プロジェクト（共有ストレージ）とすることによるチーム共同利用の実現
